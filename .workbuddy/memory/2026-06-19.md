## 2026-06-19 晚 — 全站滚动重叠问题修复 + 构建验证

### 问题诊断
用户反馈："所有页面中顶部标签/卡片在向下滚动后仍停留在原位，导致与内容拥挤打架"

**根因分析（3 类问题）：**
1. **多层 sticky 打架**：DiscoverPage / CirclePage / SearchPage 存在两个独立 `position: sticky` 元素（header + tabs），各自设置不同 `top` 值，滚动时互相重叠
2. **嵌套滚动上下文**：PostDetail / PostCreate 的 body 区域设置了 `overflow-y: auto`，与 PhoneShell 的 `.phone-screen`（唯一正确滚动容器）产生嵌套滚动
3. **弹窗滚动泄漏**：MessagePage 的 plus-menu 是 in-flow 元素，展开时推挤内容，背景滚动未锁定

### 修复方案（统一模式）

| 文件 | 问题 | 修复 |
|------|------|------|
| `HomePage.vue` | sticky header 缺少视觉分层 | 添加 `box-shadow` |
| `DiscoverPage.vue` | 双 sticky（header + tabs） | 合并为 `.sticky-header-wrap` 单层 |
| `CirclePage.vue` | 双 sticky（header + tabs） | 合并为 `.sticky-header-wrap` 单层 |
| `SearchPage.vue` | 双 sticky + 重复 result-tabs | 合并 + 删除重复 DOM |
| `PostDetail.vue` | `.detail-body` 嵌套 overflow-y:auto | 移除嵌套滚动，footer 改为 sticky |
| `PostCreate.vue` | `.create-body` 嵌套 overflow-y:auto | 移除嵌套滚动 + flex 列布局 |
| `MessagePage.vue` | plus-menu 为 in-flow 无滚动锁 | 改为 Teleport 叠加层 + store.lockPhoneScroll() |

### 关键代码模式
```css
/* 单层 sticky 包裹模式 */
.sticky-header-wrap {
  position: sticky;
  top: 0;
  z-index: 100;
  background: var(--echo-white);
  max-width: 375px;
  box-shadow: 0 1px 0 var(--echo-border), 0 4px 8px rgba(0,0,0,0.04);
}
```

function openPopup() { visible.value = true; store.lockPhoneScroll() }

## 2026-06-19 深夜 — 顶部区域滚动行为全面重构（移动端真实交互）

## 16:45 — fixed-header + page-scroll 双层结构重构

### 问题
之前的 `position: sticky; top: 0` 方案虽然标准 CSS 理论正确，但在当前 phone-shell 架构下（`.phone-body` 有 `transform: translateZ(0)` 创建固定层包含块），

### 架构决策：DOM 级双层隔离

不再让 sticky 元素与帖子流在同一个 DOM 上下文中共存，改为两层完全隔离的结构：

```
phone-body (overflow:hidden; transform:translateZ(0))
├── notch-area (48px, z-index:100)
├── phone-screen (overflow-y:auto; flex:1)
│   └── page-root
│       ├── fixed-header (position:fixed; top:48px; z-index:99)
│       │   ├── top-hero (滚动后 max-height:0 收缩隐藏)
│       │   └── channel-tabs (始终可见)
│       └── page-scroll (普通文档流)
│           ├── header-spacer (高度由 JS 动态计算)
│           ├── sub-tags
│           └── post-list
├── tabbar-area (56px)
└── home-bar (32px)
```

### 关键机制

1. **position:fixed 定位**：`.phone-body` 的 `transform: translateZ(0)` 使其成为 `position: fixed` 子元素的包含块，fixed-header 相对手机壳定位而非 viewport

// 最近聊天列表（消息搜索默认页）
const recentChats = computed(() => {
  const list = [...store.chatListData]
  const parseTime = (t) => {
    if (!t) return 0
    if (t.includes(':')) {
      const [h, m] = t.split(':').map(Number)
      return h * 60 + m
    }
    return 0
  }
  return list.sort((a, b) => {
    if ((b.unread || 0) !== (a.unread || 0)) {
      return (b.unread || 0) - (a.unread || 0)
    }
    return parseTime(b.lastTime) - parseTime(a.lastTime)
  }).slice(0, 20)
})

function onRecentChatClick(chat) {

  }
}

## 20:28 — 他人主页白屏崩溃Bug修复（watch TDZ 问题）

### 问题现象
点击他人头像进入个人主页（`/profile/:uid`）后：
1. 页面完全空白（仅底部 TabBar 可见）
2. 切换到首页/发现等其他页面**也完全空白**
3. 只有刷新浏览器才能恢复正常

### 根因分析

**`watch(contentTabs, {immediate:true})` 声明位置在 `contentTabs` computed 定义之前。**

UserProfile.vue script 执行顺序：
```
Line 235: activeTab = ref('posts')
Line 238: watch(contentTabs, fn, {immediate:true})  ← ❌ contentTabs 尚未定义！
...
Line 380: contentTabs = computed(...)                 ← 这里才定义
```

虽然 `<script setup>` 通常能处理前向引用，但 `immediate: true` 的 watcher 在 `watch()` 调用时**同步触发回调**，此时 `contentTabs` computed 可能处于未完成初始化的状态。

一旦 Vue 运行时报错 → App.vue 中 `<transition mode="out-in">` 状态机卡死 → **后续所有路由切换都渲染失败**（因为 transition 内部状态损坏），只有完整页面刷新才能重置。

### 修复方案

将 `watch(contentTabs, ...)` 从第 238 行移动到第 399 行（`contentTabs` computed 定义之后），确保 watcher 注册时 computed 已完全初始化：

```js
// ✅ 修复后：watch 在 contentTabs 定义之后
const contentTabs = computed(() => { ... })

// 注意：此 watch 必须放在 contentTabs 定义之后
watch(contentTabs, (newTabs) => {
  if (!newTabs || !newTabs.length) return
  const tabKeys = newTabs.map(t => t.key)
  if (!tabKeys.includes(activeTab.value)) {
    activeTab.value = tabKeys[0] || 'posts'
  }
}, { immediate: true })
```

- 清除 `node_modules/.vite` + `dist` 构建缓存
- 强制重新构建

## 21:06 — ChannelManage 全屏覆盖定位失效修复（紧急）

### Bug 现象
频道管理界面内容直接嵌入页面流中显示，无毛玻璃背景、无顶部导航栏、「我的频道」标题不可见。用户看到的是网格标签和推荐频道直接混在首页帖子流中。

### 根因
`.cm-overlay` 使用 `position: absolute; inset: 0;` 定位在 `#phone-screen` 上。
但 `#phone-screen` 是 `display:flex + overflow-y:auto` 容器——在此类容器中 `absolute + inset:0` **无法正确计算尺寸**，导致 overlay 退化成普通流内元素。

### 修复方案
```css
/* 修复前 */
.cm-overlay { position: absolute; inset: 0; ... }

/* 修复后 */
.cm-overlay {
  position: fixed;
  top: 0; left: 0; right: 0; bottom: 0;
  width: 375px; height: 100%; max-height: 812px;
  margin: auto;
  ...
}
```
利用 `.phone-body` 的 `transform:translateZ(0)` 创建的 fixed 包含块，确保 overlay 覆盖整个手机屏幕。
